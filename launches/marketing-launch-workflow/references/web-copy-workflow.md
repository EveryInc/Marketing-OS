# Web Copy Workflow

How Douglas runs web copy for product launches. Extracted from the Monologue website
redesign process.

## The pattern

1. Douglas sends a Notion source doc — usually a landing page redesign page with
   customer evidence, competitive notes, and rough direction.
2. Produce a positioning brief grounded in the source doc, `foundation/marketing-os`,
   and a competitor audit.
3. Douglas gives headline direction (e.g., "THINK OUT LOUD" or "turns your voice
   into X?").
4. Push back with variants organized by approach (transformation, wordplay, imperative,
   personality), with competitive context.
5. Douglas refines live — rapid-fire thread, often gives his own version and says
   "push on this."
6. Push again — with conversion best practices, competitive scan, and positioning doc
   reference.
7. Douglas may ask for a Claude prompt if he wants to iterate further outside the
   thread.
8. Final copy lands in a Google Doc or directly into Notion deliverables.

## Rules (from actual corrections)

**Always reference the positioning brief.** Douglas will ask "are you using the doc we
just made?" If the answer is no, credibility is already lost.

**Do not fall into category traps.** If the positioning says "voice app for work," do
not write headlines with "dictation." The positioning doc explicitly warns against this.

**No pseudo-profound sentences.** If a sentence tries to be clever but does not parse
logically, cut it. Douglas will flag it: "makes zero grammatical sense."

**No hype words.** Revolutionary, game-changing, reimagined — all banned per
`foundation/marketing-os`.

**When copy is weak, fix it immediately.** Do not explain why you wrote it that way.
Produce better options.

**Provide variants organized by approach.** Do not send a flat list — group by the
type of move each option makes.

**Competitive context matters.** Know what every relevant competitor says in their hero
copy. Do not accidentally write their copy.

## Hero copy structure

The hero is the biggest conversion lever. Based on the Monologue process:

```
Hed: [Short, evocative, behavior-first — not feature-first]
Sub: [Grounds the hed — tells you what the product actually does]
     [Permission/normalization + specific inputs + clean output]
     [Scope claims: "Any language. Anywhere."]
```

What works: normalizing the behavior, listing specific input types (notes, calls,
meetings, ideas), concrete scope claims.

What does not work: category labels ("AI dictation"), speed claims competitors already
own, vague feature descriptions, sentences that are technically correct but do not say
anything real.

## Page-by-page copy flow

Standard structure based on best-practice analysis:

1. **Hero** — Category plus promise. Above fold. One hed, one sub, one CTA.
2. **Social proof band** — Trust. Immediately below hero. Logos or quotes from
   recognizable names.
3. **Feature sections** — Evidence. Bold-label descriptions with supporting copy. Show,
   do not tell, where possible.
4. **Use cases / personas** — Identification. "People like you use this."
5. **Product demo / screenshots** — Proof. The product actually working.
6. **Pricing** — Commitment. Clear, simple, no cognitive load.
7. **Final CTA** — Action. Repeat the core promise, single button.

## Subheadline workshop process

When Douglas gives a subheadline direction and says "push on this":

1. Analyze what is working and why (name the specific moves).
2. Compare against competitor hero copy.
3. Identify two to three specific things to tighten (e.g., drop the "AI" label, sharpen
   the output promise, swap "any time" for "anywhere").
4. Produce a pushed version.
5. If appropriate, produce a Claude prompt Douglas can use for further iteration.
